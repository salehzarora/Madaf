## القسم 2 — نطاق التجربة (Pilot Scope)

نطاق أول دفعة **مُحافِظ ومُراقَب**:

- مورّد/مستودع تجريبي **واحد** (tenant واحد).
- **مالك واحد** (`owner`).
- اختياريًا **مدير واحد** (`admin`).
- اختياريًا **مندوب مبيعات واحد** (`sales_rep`).
- عدد **محدود** من المحلات/الزبائن الحقيقيين.
- كتالوج منتجات **مبدئي محدود**.
- سير عمل الطلبيات والمخزون الاعتيادي.

### مدعوم اليوم في التجربة

| القدرة | المسار / الآلية |
|---|---|
| تسجيل دخول المورّد (OTP هاتف كطريقة أساسية) | `/login` |
| إنشاء المورّد لأول مرة | `/onboarding` |
| كتالوج المنتجات والشركات المصنّعة | `/admin/products` ، `/admin/manufacturers` |
| المخزون وسجل حركة المخزون | `/admin/inventory` ، `/admin/inventory/movements` |
| المحلات (الزبائن) وإدارتها | `/admin/customers` |
| التسجيل الذاتي للمحلات عبر رابط | `/admin/customers/signup` → `/join/<token>` |
| الطلبيات وتغيير حالتها | `/admin/orders` |
| روابط الطلب الخاصة للزبون (بدون تسجيل دخول) | `/shop/<token>` |
| روابط العرض للزوّار | `/showcase/<token>` |
| تصدير CSV للطلبيات، المنتجات، وحركة المخزون | من الجداول المعنية |
| سجل النشاط (Activity Log) للزبائن والطلبيات | داخل صفحة الزبون / الطلب |

### مؤجَّل / خارج نطاق التجربة

- **الفواتير الضريبية القانونية، الترقيم القانوني، أي مزوّد ضريبي** — يجب أن تبقى **مُطفأة**.
  صفحات المستندات (`/admin/documents`) هي **مسودّات** فقط ولا تُقدَّم كفاتورة ضريبية صادرة قانونيًا.
- **المدفوعات** — لا توجد مرحلة مدفوعات في المنتج.
- **الرسائل النصية SMS كميزة منتج** — غير مشمولة (خارج تسجيل الدخول عبر OTP).
- **دعوات الفريق عبر البريد** — تعمل فقط إذا فُعِّل `MADAF_EMAIL_AUTH_ENABLED` عمدًا (انظر «علامة دعوات الفريق»).
- صفحة الإعدادات هي **إعدادات المنشأة** + **إعدادات الضريبة** فقط؛ لا توجد صفحة «إعدادات» عامة أخرى.

> لا تُنشئ أو تفترض وحدات غير مدعومة (لا مدفوعات، لا فواتير قانونية مفعّلة، لا SMS كميزة، لا أدوات تحليلات، لا نسخ احتياطي تلقائي مُضاف في هذه المرحلة).

## القسم 3 — مسار الإعداد والانضمام (Onboarding)

هناك **مساران منفصلان** يجب عدم الخلط بينهما:

### (أ) انضمام المورّد نفسه (إنشاء الـ tenant)

1. المورّد يفتح `/login` ويسجّل الدخول (**OTP هاتف** كطريقة أساسية؛ البريد/كلمة المرور احتياطي مخفي — انظر «علامة دعوات الفريق»).
2. مستخدم مسجّل الدخول **بلا عضوية** يُحوَّل تلقائيًا إلى `/onboarding` (العنوان: **«إنشاء مورّد»**).
3. يضغط **«إنشاء المورّد»** → يصبح **`owner`** لهذا المورّد.
4. **تحقّق من سياق المستأجر (tenant context):** بعد الدخول يجب أن تعرض لوحة التحكم بيانات هذا المورّد فقط، والدور المعروض بجانب **«تسجيل الخروج»** هو **«مالك»**.
5. تعيين الأدوار وإضافة الفريق: `/admin/team` (العنوان: **«الفريق»**).
6. الإعدادات الأولية: **«إعدادات المنشأة»** (`/admin/settings/business`) و**«إعدادات الضريبة»** (`/admin/settings/tax`).

### (ب) انضمام المحل/الزبون (التسجيل الذاتي — «join»)

> **مهم:** الموافقة على طلب تسجيل محل **تُنشئ زبونًا (محل/Customer)** داخل مستأجر المورّد،
> وليست إنشاء مستأجر جديد. إنشاء المستأجر يخصّ المورّد فقط (المسار أ).

1. **طلب رابط تسجيل:** المالك/المدير يفتح `/admin/customers/signup` («**تسجيل محلات جديدة**») → **«إنشاء رابط تسجيل»** (يظهر الرابط **مرة واحدة فقط** — انسخه فورًا). الرابط بصيغة `/join/<token>`.
2. **المحل يُرسل بياناته:** يفتح `/join/<token>` (بدون تسجيل دخول)، ويملأ استمارة **«تسجيل محل جديد»** (اسم المحل مطلوب؛ جهة الاتصال/الهاتف/البريد/المدينة/العنوان/ملاحظات اختيارية)، ثم **«إرسال بيانات المحل»**. رسالة النجاح: **«تم إرسال طلب تسجيل المحل»**.
3. **مراجعة المدير:** الطلب يظهر في جدول **«طلبات محلات جديدة»** بحالة **«بانتظار المراجعة»**. القائمة **مُقسّمة إلى صفحات** (50 لكل صفحة) والأقدم يبقى قابلًا للوصول عبر **«السابق/التالي»**.
4. **تحذير التشابه (Duplicate):** إذا وُجد محل بنفس **الاسم أو الهاتف**، يظهر تنبيه **«يوجد محل بنفس رقم الهاتف أو الاسم»** مع خيار **«فتح المحل»** لمعاينة المطابق.
5. **تجاوز صريح للتشابه (فقط حيث يُدعم):** الضغط على **«الموافقة رغم التشابه»** يتخطّى فحص التشابه ويكمل القبول. هذا خيار **صريح لكل عملية**؛ لا يُخزَّن أي شيء دائم عن التجاوز.
6. **القبول → إنشاء الزبون:** الضغط على **«قبول»** يُنشئ سجل زبون (Customer) بمصدر `signup`، ويُسجَّل حدث نشاط **«تم إنشاء زبون»** — كل ذلك في معاملة واحدة. أو **«رفض»** لإنهاء الطلب بالرفض.

> لا تُوافق على أي طلب حقيقي أثناء إعداد هذا الدليل أو أثناء الاختبار. استخدم سجلًّا تجريبيًا مُصرَّحًا به فقط.

### علامة دعوات الفريق — `MADAF_EMAIL_AUTH_ENABLED`

- **ما الذي يتحكّم به:** تفعيل **تسجيل الدخول/التسجيل عبر البريد وكلمة المرور** (في الواجهة **وفي الخادم معًا**). متغيّر **خادمي فقط** (never `NEXT_PUBLIC`)؛ القيمة المفقودة/الفارغة = `false`.
- **السلوك في الإنتاج (OTP هاتف كطريقة أساسية):**
  - غير مضبوط / `false` → البريد وكلمة المرور **مُعطَّلان تمامًا** (تُخفى في الواجهة وتُرفض في الخادم).
  - `=true` → البريد وكلمة المرور **مسموحان**.
- **علاقته بدعوات الفريق:** الدعوة عبر البريد تتطلّب عند **القبول** أن يطابق بريد الحساب بريد الدعوة. الحساب المُنشأ بالهاتف فقط **بلا بريد**، فلا يستطيع قبول دعوة بريدية. لذلك في إنتاج يعتمد الهاتف، **لا تعمل دعوات الفريق البريدية إلّا إذا فُعِّل هذا المتغيّر**.
- **هل التجربة تحتاجه؟** **التسجيل الذاتي للمحلات لا يحتاجه إطلاقًا** (مجهول المصدر، بلا تسجيل دخول).
  - إن لم تعتمد التجربة على دعوات الفريق البريدية → **يجوز إبقاء المتغيّر كما هو دون تغيير**.
  - إن كانت التجربة تحتاج دعوات فريق بريدية → **تفعيله والتحقّق منه قرار مشغّل صريح ومنفصل قبل الإطلاق**.
- **لا تغيّر هذا المتغيّر تلقائيًا. لا تعدّل إعدادات Vercel في هذا الدليل.**

> ⚠️ **تحذير قفل تسجيل الدخول:** في بيئة تعتمد الهاتف **بدون مزوّد SMS** و`MADAF_EMAIL_AUTH_ENABLED=false` وبدون مسار OTP التجريبي، **لا توجد طريقة لتسجيل الدخول إلى الإدارة**. تأكّد من توفّر طريقة دخول واحدة صالحة على الأقل قبل الإطلاق.

## القسم 4 — الأدوار والوصول (Roles & Access)

الأدوار الثلاثة: `owner` (**مالك**) · `admin` (**مدير**) · `sales_rep` (**مندوب مبيعات**).
**حدود الصلاحية الفعلية هي التحقّق في الخادم (RPC + RLS)، لا ظهور عناصر القائمة.**

| القدرة | owner | admin | sales_rep |
|---|:--:|:--:|:--:|
| قراءة الكتالوج/المخزون (نفس المورّد) | ✓ | ✓ | ✓ |
| إنشاء/تعديل المنتجات والمخزون والشركات والأقسام | ✓ | ✓ | ✗ |
| قراءة المحلات (الزبائن) | الكل | الكل | **المُسندة فقط** |
| إنشاء/تعديل المحلات | ✓ | ✓ | ✗ |
| إنشاء طلبية | لأي محل | لأي محل | **لمحل مُسنَد فقط** |
| قراءة الطلبيات وتفاصيلها | الكل | الكل | **طلبيات المحلات المُسندة فقط** |
| تغيير حالة الطلبية | ✓ | ✓ | ✗ |
| إنشاء/إلغاء روابط الزبون الخاصة | ✓ | ✓ | ✗ (لا يرى حتى بيانات الروابط) |
| عرض قائمة الفريق (`/admin/team`) | ✓ | ✓ | ✗ |
| إرسال/إلغاء دعوات (أدوار: admin أو sales_rep فقط) | ✓ | ✓ | ✗ |
| إسناد المحلات للمندوبين | ✓ | ✓ | ✗ |
| تغيير دور عضو / إزالة عضو | ✓ | ✗ | ✗ |
| الترقية إلى مالك / إنزال مالك | ✓ | ✗ | ✗ |

قيود إضافية مفروضة في قاعدة البيانات (وليست في الواجهة فقط):
- دور **`owner` يُمنَح حصريًا** عبر `promote_tenant_owner` (لمالك فقط) — **لا عبر الدعوات**.
- **لا يمكن للعضو تغيير دور نفسه**، ولا يمكن أن يهبط المستأجر إلى **صفر مالكين** (حماية آخر مالك).
- **مندوب المبيعات** يطلب فقط لمحل **مُسنَد**؛ **لا يوجد رجوع إلى «كل المحلات»**.

**حماية المسارات الحسّاسة (تحقّق مباشر على مستوى الصفحة):** إنشاء/تعديل المنتج
(`/admin/products/new`، `/admin/products/[id]/edit`) وإنشاء/تعديل الزبون
(`/admin/customers/new`، `/admin/customers/[id]/edit`) وصفحة **«الفريق»** كلها
تُعيد **404** لمندوب المبيعات **قبل تحميل أي نموذج أو بيانات** — بالإضافة إلى بوابة الـ RPC والـ RLS.

> **ظهور عناصر القائمة ليس هو حدّ الأمان.** الحماية الحقيقية = بوابة الخادم (`notFound()`) + بوابة `authorize_tenant([owner,admin])` في الـ RPC + سياسات RLS. ثلاث طبقات مستقلة.

**وصول الزبون المجهول (روابط `/shop/<token>`):** الزبائن **لا يسجّلون الدخول**. الرمز
(token) نفسه هو بيانات الاعتماد، لذا الصفحة `noindex`. المجهول لا يملك أي وصول مباشر
للجداول، ولا يمكنه تحديد المستأجر/الزبون/الأسعار/الإجمالي؛ يستطيع فقط جلب كتالوج رابطه
وإرسال طلبية عبر دالتين آمنتين محدّدتين. الرمز الخام **لا يُخزَّن أبدًا** (يُخزَّن هاش SHA-256 فقط).

## القسم 5 — إعداد أول مستأجر (First-Tenant Setup)

استخدم إعدادات وميزات **حقيقية فقط**. لا تُنشئ بيانات تجريبية على المستضاف الإنتاجي.

- [ ] **هوية المنشأة:** `/admin/settings/business` («**إعدادات المنشأة**») — الاسم، الشعار، بيانات الاتصال (تظهر على المستندات).
- [ ] **المنطقة الزمنية للمورّد:** تُضبط ضمن **«إعدادات المنشأة»**؛ تؤثّر على عرض التواريخ/الأوقات في كل الشاشات بحسب منطقة المورّد.
- [ ] **عضوية المالك/المدير:** موجودة وصحيحة في `/admin/team`.
- [ ] **عضوية مندوب المبيعات (إن استُخدم):** مُضافة، **ومُسندة لمحلات محدّدة**.
- [ ] **كتالوج المنتجات:** `/admin/products` → **«إضافة منتج»**؛ أدخِل الأسماء بالعربية/العبرية/الإنجليزية والعبوات والأسعار.
- [ ] **خيار متابعة المخزون لكل منتج:** خانة **«متابعة المخزون لهذا المنتج»** في نموذج المنتج — **مُطفأة افتراضيًا**.
- [ ] **المحلات (الزبائن):** أضِف عبر `/admin/customers` → **«إضافة محل»**، أو عبر التسجيل الذاتي (القسم 3-ب).
- [ ] **رابط الزبون الخاص:** من `/admin/customers` أنشئ رابط طلب، وافتح `/shop/<token>` للتأكّد.
- [ ] **سير الطلبيات:** أنشئ/استقبل طلبية وتأكّد من عرض الحالة والمراجع.
- [ ] **اللغة والعرض:** العربية/العبرية (RTL) والإنجليزية (LTR) تظهر بشكل صحيح.
- [ ] **حالة التفعيل/التعطيل للمحل:** حيث تُدعم (المحل المعطّل يحتفظ بتاريخه وتذهب روابطه إلى وضع خامل).

### قاعدة المخزون (مهمة ودقيقة)

| الحالة | ما يظهر للزبون | قابل للطلب؟ |
|---|---|---|
| منتج **بلا سجل مخزون** (متابعة المخزون مُطفأة) | **«متوفر»** (In stock) | **نعم** — يُعامَل كمتوفّر (غير متتبَّع) |
| منتج **متتبَّع بكمية 0** أو أقل | **«غير متوفر»** (Out of stock) | **لا** — زر الإضافة للسلة مُعطَّل |
| منتج متتبَّع بكمية موجبة **أقل من الحدّ** | **«مخزون منخفض»** (Low stock) | نعم — ما زال قابلًا للطلب |

> باختصار: «لا سجل مخزون» ≠ «مخزون صفر». الأول **متوفّر**، والثاني **غير متوفّر**. فعّل متابعة المخزون فقط للمنتجات التي تريد أن يُخفيها النظام تلقائيًا عند نفادها.

### القسم 5-ب — المستأجر التشغيلي للفحص `PILOT-SMOKE`

> ⛔ **لا تُنفَّذ طلبيات فحص الدخان داخل مستأجر النشاط الحقيقي.**
> **التطبيق لا يوفّر أي مسار لحذف طلبية** (CONFIRMED FROM REPOSITORY). أي طلبية فحص تُنشأ في مستأجر
> النشاط تبقى **سجلًّا دائمًا** يظهر في تصديراته ولوحته وأرقام أسبوعه الأول.

**القاعدة المعتمَدة:**

| البند | القرار |
|---|---|
| عدد مستأجري الفحص | **واحد فقط**، اسمه `PILOT-SMOKE`، يملكه المشغّل |
| متى يُنشأ | **مرّة واحدة**، خلال مرحلة التحقّق المستضاف المُصرَّح بها لاحقًا — **لا يُنشأ أثناء المرحلة التوثيقية** |
| إعادة الاستخدام | **يُعاد استخدام نفس المستأجر** لكل فحص لاحق |
| مستأجر جديد لكل نشر | ❌ **ممنوع** — يُراكم مستأجرين مهجورين بلا فائدة |
| المحتوى | منتجات/زبائن/طلبيات/مخزون/صور/مستندات **قابلة للإهمال فقط** |
| بيانات حقيقية | ❌ **لا بيانات زبائن حقيقية ولا معلومات شخصية إطلاقًا** |
| حساب المشغّل | **منفصل** عن حساب مالك النشاط الحقيقي |
| مقاييس التجربة | **تُستثنى** بيانات `PILOT-SMOKE` من كل أرقام النشاط والتقارير |

**تسمية السجلات داخل `PILOT-SMOKE`:**

```
PILOT-SMOKE-YYYYMMDD-<PURPOSE>
```

مثال الشكل: `PILOT-SMOKE-20260720-ORDER-RETRY` · `PILOT-SMOKE-20260720-IMAGE-SIGNING`.

> **مهم — لا تَعِد بحذف:** طلبيات الفحص **سجلات تحقّق دائمة**. لا يوجد إجراء حذف في التطبيق، ولا
> يجوز اختراع واحد، ولا يجوز حذف صفوف من قاعدة البيانات كإجراء «تنظيف». البادئة والتاريخ هما
> وسيلة التمييز — لا الحذف.

## القسم 7 — فحوص سلامة البيانات (Data-Integrity Checks)

بلغة المشغّل:

- **تصدير الطلبيات CSV:** لا يتوقّف صامتًا عند 1000 صف؛ يُطبَّق حدٌّ منطقي أقصى؛ عند فشل الترقيم **يفشل بأمان** (لا يُرجِع بيانات جزئية مُضلِّلة كأنها كاملة).
- **لوحة التحكم:** المقاييس مُحسوبة في قاعدة البيانات؛ **الطلبيات التي تتجاوز 1000 تبقى محتسَبة**؛ عدد طلبات التسجيل المعلّقة **دقيق ومُحصى فقط** (بلا نقل أي بيانات شخصية لمجرّد الحساب).
- **حركة المخزون:** تُجلب فقط أرقام الطلبيات المُشار إليها في الصفحة الحالية؛ مراجع الطلبيات الأقدم تبقى قابلة للحلّ.
- **طلبات التسجيل:** القائمة مُقسّمة إلى صفحات؛ الطلبات الأقدم تبقى قابلة للوصول؛ **القبول والرفض حالتان نهائيتان متنافيتان**؛ قبول متزامن مزدوج يُنشئ **زبونًا واحدًا كحدٍّ أقصى**؛ قبول مقابل رفض متزامن يسمح **بفائز نهائي واحد فقط**.
- **المستندات (PILOT-OPS-AUDIT-008):** قائمة المستندات (`/admin/documents`) **مُقسّمة إلى صفحات ومحدودة** — لا قراءة غير محدودة تسقط صفوفًا صامتًا عند سقف PostgREST، والأحدث أولًا، مع إثراء محدود بالصفحة لرقم الطلب واسم الزبون فقط. **التقسيم هو ترقيم إزاحة (offset) محدود بترتيب ثابت (`created_at DESC, id DESC`)، وليس keyset** — الحدّ هو ما يمنع الاقتطاع؛ وقد يُزيح إدراج/حذف متزامن صفًّا واحدًا بين صفحتين متجاورتين، ويُصحِّح التحديثُ العرضَ الحالي. (تحويلها إلى keyset تغيير مستقبلي منفصل.)
- **تزامن الطلبيات (PILOT-OPS-AUDIT-008):** تأكيد/تعديل/إلغاء طلبيتين متزامنتين تمسّان **نفس المنتجات** لم يعد يُنتِج **تعطّلًا (deadlock)** — تُقفل صفوف المخزون بترتيب `product_id` تصاعدي موحَّد. الأرصدة النهائية تبقى صحيحة (حجز مرة واحدة، استرجاع مرة واحدة).
- **تكرار تقديم الطلبية محسوم في قاعدة البيانات (PILOT-OPS-AUDIT-008-FIX1):** لكل تقديم طلبية **مفتاح تقديم UUID واحد** تُنشئه الواجهة وتُعيد استخدامه عبر المحاولات (بما فيها بعد فشل شبكي غامض). قاعدة البيانات هي بوّابة عدم التكرار الموثوقة: **إعادة إرسال مطابقة تُعيد الطلبية نفسها** (لا طلبية/سطور/رقم/مرجع/حدث `order.created` ثانٍ)، و**إعادة استخدام المفتاح مع طلبية مختلفة تفشل بأمان (MDF40)** فتُبقي الواجهة السلّة وتعرض «ابدأ محاولة جديدة». يسري على **الكل**: مسار المورّد المسجَّل، رابط المتجر المُرمَّز، وطلب الزائر من العرض. **لا يُخزَّن رمز خام**، **لا تُوثَق مجاميع/ضرائب من العميل** (كلها تُحسب في الخادم)، **ولا يظهر المفتاح في الواجهة أو في بيانات التدقيق**. المسار القديم غير المُحصَّن (بلا مفتاح) **أُزيل** — لا التفاف تشغيلي.
- **بقاء مفتاح التقديم عبر التحديث/إعادة التركيب (PILOT-OPS-AUDIT-008-FIX2):** مفتاح مسار المورّد المسجَّل يبقى وفق تخزين سلّته (localStorage). أمّا **رابط المتجر المُرمَّز وطلب الزائر من العرض** فيبقى مفتاحهما في **`sessionStorage`** (مربوط بالتبويب/الجلسة، ومساحته مُسمّاة بـ **القناة + بصمة SHA-256 للرمز — بلا رمز خام**). لذلك **يبقى أمان إعادة المحاولة سليمًا عبر تحديث الصفحة/إعادة التركيب داخل نفس التبويب**: الاستجابة الغامضة **تُبقي** المفتاح؛ النجاح المؤكَّد **يمسحه**؛ و«ابدأ محاولة جديدة» **يُدوّره**. **علامتا تبويب منفصلتان = محاولتان منطقيّتان منفصلتان** (لا إزالة تكرار عبر التبويبات — قيد القاعدة يمنع أي تكرار حقيقي لأي مفتاح). إن تعذّر تخزين المتصفّح بأمان، **لا تُرسَل الطلبية** (فشل مُغلق): تُعرض رسالة تجهيز مترجمة وتُحفَظ السلّة، دون ادّعاء أن الطلبية أُرسلت. **قاعدة البيانات تبقى بوّابة عدم التكرار الموثوقة.**
- **تسلسل التسجيل الذاتي (PILOT-OPS-AUDIT-008-FIX1):** تسجيلان ذاتيّان متزامنان لنفس المستخدم يُنتجان **مستأجرًا واحدًا فقط**؛ صف `auth.users` الخاص بالمستخدم هو حدّ التسلسل (`FOR UPDATE`)، والخاسر يتلقّى خطأ «عضو بالفعل» القائم. مستخدمون مختلفون لا يتعارضون.

### الأحداث المؤكَّدة — الفريق (الفئة: «الفريق») — PILOT-OPS-AUDIT-003

أُضيفت في **المرحلة 3**. مُثبَتة محليًا (pgTAP 88 + وحدات + واجهة + **فحوص تزامن بجلستين

داخل صفحة الفريق** (`/admin/team`، `entity_type=team`، على مستوى المستأجر كله). مرئية لـ
**المالك/المدير فقط** (RLS + بوابة المسار)؛ **المندوب لا يراها**. لا صفحة سجل نشاط عامّة، ولا
عنصر تنقّل جديد، ولا صفحة تفاصيل عضو.

| الحدث | متى | الحساسية | التسمية العربية |
|---|---|---|---|
| `team.member_invited` | إدراج دعوة ناجح (دور: مدير/مندوب) | متوسطة | «تمت دعوة عضو» |
| `team.invitation_revoked` | انتقال فعلي لدعوة معلّقة إلى «ملغاة» | متوسطة | «أُلغيت الدعوة» |
| `team.member_joined` | قبول دعوة ينشئ العضوية ويُعلّم الدعوة مقبولة | متوسطة | «انضمّ عضو» |
| `team.role_changed` | تغيير دور فعّال (تحديث/ترقية/تخفيض ملكية) — `from_role`→`to_role` | **عالية** عند مساس الملكية، وإلا متوسطة | «تغيّر الدور» |
| `team.member_removed` | حذف عضوية نهائي (hard delete) — تُلتقط البيانات **قبل** الحذف | **عالية** عند إزالة مالك، وإلا متوسطة | «تمت إزالة عضو» |

- **هوية العضو المتأثِّر (`target_email`):** مفتاح موحَّد واحد على **كل** حدث، مُطبَّع
  (`lower(trim)`، ≤254)، يُحلّ **داخل** المُنتِج من مصدر موثوق (`auth.users.email` للعضو /
  بريد الدعوة المُقفَل) — **ليس** معاملًا عامًّا في أي RPC ولا قيمة يرسلها العميل. يبقى
  **قابلًا للقراءة بعد الإزالة** (لأن العضوية تُحذف)، فلا حاجة لبحث هوية ثانٍ ولا UUID خام في
  الواجهة. تحليل تسمية **الفاعل** فقط عبر نفس الـ RPC المحدود بالصفحة.
- **حماية التزامن:** الدعوة تُقفَل بالصف (`FOR UPDATE`) في القبول والإلغاء (تسلسل قبول↔إلغاء؛
  الإلغاء المتكرّر وإعادة القبول بلا حدث مكرّر). الـ RPCات الحسّاسة للملكية تقفل **مجموعة
  المالكين + الهدف** بترتيب `user_id` تصاعديًّا (ترتيب قفل موحَّد ضدّ التعطّل)، ثم تُشتقّ عدّ
  المالكين والدور الحالي من صفوف مُقفَلة — **لا يمكن أن يُترك المستأجر بلا مالك**.
- **بوابات عدم-التكرار:** طلب دور مطابق للحالي **لا** يُحدِّث ولا يُصدر حدثًا؛ إلغاء دعوة
  ملغاة أصلًا **لا** يُصدر حدثًا ثانيًا. الفاعل دائمًا `auth.uid()` (لا يُقبل فاعل من العميل).
- **الأسرار:** لا رمز/تجزئة/معاينة رمز/رابط قبول/JWT/جلسة/كلمة مرور في البيانات الوصفية
  (قائمة مفاتيح مغلقة لكل حدث ترفض ما عداها).
- **مقصود ومؤجَّل:** ما زال يُسمح بدعوتين معلّقتين لنفس البريد (سلوك مؤجَّل، كل واحدة حدثها).
  **إبطال الجلسة/JWT** عند الإزالة **خارج النطاق**. تسجيل المستأجر الجديد (`create_tenant_with_owner`)
  وتسجيل المحلّات منفصلان (ليسا تدقيق فريق).

|---|---|---|---|

- **قيم آمنة مقابل مفاتيح فقط:** يُخزَّن قبل/بعد **الكامل** فقط للحقول الآمنة غير الحسّاسة —

  Europe/London`، لا إزاحة). كل حقل حسّاس (الأسماء/الاسم القانوني/رقم الشركة/أرقام التسجيل/الهاتف/
  البريد/العناوين/ملاحظات الجاهزية/الشعار) يُسجَّل **كمفتاح تغيّر فقط بلا قيمة** — يعرف المشغّل
  **أن** الحقل تغيّر ومَن ومتى، دون بقاء القيمة الحسّاسة في السجل الدائم.
- **قفل تسلسلي مشترك:** الثلاث RPCات تقفل **صف `public.tenants` المضمون** أولًا (`FOR UPDATE`)، ثم
  تلتقط الحالة السابقة المقفلة، تُطبّع، تحسب الفروق (`IS DISTINCT FROM` على القيم المخزَّنة فعلًا)،

  سباقات ولا تعطّل. حفظ لا يُغيّر شيئًا فعليًّا (بما فيه فروق المسافات/الحالة) = **لا UPDATE ولا حدث**،
  ويُعاد الصف/القيمة كما هي.
- **إغلاق مسار التعديل المباشر:** أُزيلت صلاحية `UPDATE` المباشرة على `public.tenants` من دور
  `authenticated` وسياستها، فأصبحت الـ RPCات (SECURITY DEFINER) مسار الكتابة **الوحيد** — التدقيق
  غير قابل للتجاوز. القراءة (`SELECT`) والتسجيل (`create_tenant_with_owner`) ومشغّل التحقق من المنطقة
  الزمنية سليمة. تغيير المنطقة الزمنية **لا يُعيد كتابة** أي طابع زمني مخزَّن (UTC ثابت).

`entity_type=sales_rep_assignment`، على مستوى المستأجر). مرئية لـ **المالك/المدير فقط** (RLS +
بوابة المسار)؛ **المندوب لا يراها — ولا يرى حتى تعييناته هو**.

| الحدث | متى | الحساسية | التسمية العربية |
|---|---|---|---|
| `sales_rep_assignment.created` | إسناد يدوي لمحل إلى مندوب (زوج جديد) | متوسطة | «تم تعيين مندوب للمحل» |
| `sales_rep_assignment.removed` | إزالة زوج إسناد (يدوي / إزالة عضو / تغيير دور / انضمام) | متوسطة | «أُلغي تعيين مندوب» |

- **إدارة يدوية (المالك/المدير فقط):** الإسناد/الإلغاء عبر RPCات SECURITY DEFINER؛ **المندوب
  ممنوع من الإدارة**. إسناد مكرَّر أو إلغاء لزوج غير موجود = **لا تغيير ولا حدث** (تُعاد `void`).
- **لقطات آمنة:** كل حدث يحمل `rep_user_id` + `rep_email` (مُطبَّع، ≤254) + `customer_name` (≤200)
  + `source` — لا غير. `rep_user_id` **لا يُعرض أبدًا** كنص خام؛ الصف يبقى مقروءًا بعد إزالة العضو أو
  حذف المحل. **لا** هاتف/بريد/عنوان/أرصدة/طلبات/ملاحظات/رمز/JWT/جلسة.
- **قيم `source`:** الإنشاء = `manual` فقط؛ الإزالة = `manual` | `member_removed` | `role_changed`
  | `member_joined` (تُعرض بسطر مصدر مُترجَم يوضّح **سبب** الإزالة).
- **إغلاق الوصول العالق (أمني):** `can_access_customer` / `can_access_order` تشترطان الآن للمندوب
  **معًا** وجود إسناد مطابق **وعضوية `sales_rep` حالية**. فعضو مُزال، أو عضو تغيّر دوره بعيدًا عن
  المندوب، أو صف إسناد يتيم قديم = **مرفوض فورًا** بلا حاجة لتحديث JWT/الجلسة. المالك/المدير يبقيان
  على مستوى المستأجر.
- **لا إعادة تفعيل صامتة:** خروج الدور (مندوب→مدير/مالك)، ودخول الدور (مدير/مالك→مندوب)، وإزالة
  العضو، والانضمام/إعادة الانضمام — كلها **تُطهِّر تعيينات الهدف في المعاملة نفسها** (حدث إزالة واحد
  لكل صف بالمصدر المناسب) **قبل** تغيير الدور/العضوية وحدث الفريق الواحد القائم. فالعضو
  الداخل/العائد **لا يرث شيئًا**، وكل إسناد جديد صريح (حدث إنشاء). لا يُصدر تغيير الدور/الانضمام أي
  حدث إنشاء.
- **ترتيب قفل عالمي (بلا جمود):** قفل العضوية/الدعوة أولًا ← أقفال المحلّات تصاعديًّا بـ `customer_id`
  ← تعديل الإسناد ← أحداث الإسناد ← تعديل الدور/العضوية ← حدث الفريق أخيرًا. أُثبِت محليًّا بأربع فحوص
  تزامن بجلستين (سباق نفس الزوج، إسناد ضد خروج دور، إسناد ضد إزالة عضو، حذف محل ضد إسناد) بلا جمود.
- **حدّ حذف المحل:** حذف محل **يُسقِط** صف الإسناد عبر مفتاح خارجي (FK) **بلا حدث إسناد** — يمثّله
  تدقيق دورة حياة الزبون. **لا حذف جَملي على مستوى الترحيل** للصفوف القديمة (تبقى خاملة عبر
  المُسنِدَين المُحصَّنين حتى يُطهِّرها مسار دورة حياة وقت التشغيل).

### الأحداث المؤكَّدة — قرار طلب تسجيل المحل (الفئة: «تسجيلات المحلات») — PILOT-OPS-AUDIT-006

أُضيفت في **المرحلة 6**. مُثبَتة محليًا (pgTAP 80 + وحدات + واجهة + **خمس فحوص تزامن بجلستين
حقيقيتين بلا أي جمود**)؛ **لم تُنشَر بعد ولم تُتحقَّق في الإنتاج**. **هذه طلبات تسجيل محلات/زبائن
داخل مستأجر قائم** (المحل المُحتمَل يُقدّم عبر رابط مُرمَّز — بلا تسجيل دخول — ويراجعها **مالك/مدير
المستأجر**). القبول يُنشئ **زبونًا** (customers)، لا مستأجرًا ولا عضوية مالك؛ **لا توجد تهيئة مستأجر
أو تسجيل منصّة هنا** (`create_tenant_with_owner` مسار onboarding ذاتي منفصل خارج النطاق). تظهر كـ
**قسم «نشاط طلبات تسجيل المحلات» للقراءة فقط أسفل قائمة المراجعة** في
`/admin/customers/signup`. مرئية لـ **المالك/المدير فقط**؛ **المندوب/غير العضو/المستأجر الآخر/الزائر
المجهول لا يرونها**.

| الحدث | متى | الحساسية | التسمية العربية |
|---|---|---|---|
| `customer_signup_request.approved` | القبول ينقل الطلب PENDING → APPROVED | متوسطة | «تم قبول طلب تسجيل المحل» |
| `customer_signup_request.rejected` | الرفض ينقل الطلب PENDING → REJECTED | متوسطة | «تم رفض طلب تسجيل المحل» |

- **بلا حدث للتقديم:** الإرسال مجهول (لا فاعل)، وصف الطلب نفسه هو سجل التقديم — لا حدث تدقيق له.
- **القبول يُصدر حدثين** في المعاملة نفسها: `customer.created` (بمصدر `signup`، مُبقًى كما هو) **و**
  `customer_signup_request.approved` (على كيان الطلب). لا يُستبدل أحدهما بالآخر.
- **لقطات آمنة:** كل حدث يحمل `business_name` (اسم المحل، مُشذَّب، ≤200) فقط؛ القبول يضيف
  `resulting_customer_id` (لبناء رابط آمن للزبون — **لا يُعرض كنص خام**). **لا** بريد/هاتف/عنوان/مدينة/
  ملاحظات المُقدِّم، ولا رمز/JWT/جلسة/سر. **لا يوجد حقل سبب رفض** حاليًا.
- **سلوك الحالة النهائية والتزامن مُصان (C2):** القبول يقفل صف الطلب (FOR UPDATE) ويُعيد فحص
  «قيد الانتظار» ثم يطالب شرطيًّا؛ الرفض مطالبة شرطية ذرّية (RETURNING تُبوِّب الحدث). قيد
  `approved+rejected` معًا يبقى ممنوعًا. **قرار نهائي واحد فقط** لكل طلب؛ الأحداث تطابق الفائز؛
  محاولة مكرّرة/متسابقة → الخطأ القائم (22023) بلا حدث. الكل يتراجع معًا عند أي فشل.
- **حدود الخصوصية:** لا تدخل بيانات اتصال المُقدِّم في `audit_events`؛ التاريخ مرئي للمالك/المدير فقط.

### فجوات مؤكَّدة (مُدقَّقة، لا افتراضات)

المُدقَّق يغطّي **الزبائن والطلبيات والمنتجات والمخزون (الإعداد/التهيئة) والفريق (دعوة/قبول/
إلغاء/تغيير دور/نقل ملكية/إزالة) وإعدادات المنشأة/المنطقة الزمنية/الضريبة**. العمليات التالية **بلا
حدث سجل نشاط**:

| العملية | الحالة |
|---|---|
| تغيير إعدادات المنشأة / المنطقة الزمنية / الضريبة | **مُغطّاة الآن** (PILOT-OPS-AUDIT-004، أعلاه) |
| تعديل مخزون يدوي بشري | يبقى في **سجل حركة المخزون** (`/admin/inventory/movements`) بالكمية والسبب والملاحظة و**الفاعل الآن** — **بلا تكرار في `audit_events`** (قرار مقصود، ليس فجوة) |
| حركات المخزون المدفوعة بالطلبات | تُوثَّق عبر **تدقيق الطلب** + `order_status_history` + سجل الحركة — **بلا تكرار في المخزون** |
| تغييرات الفريق (دعوة/قبول/إلغاء/تغيير دور/نقل ملكية/إزالة) | **مُغطّاة الآن** (PILOT-OPS-AUDIT-003، أعلاه) |
| إسناد/إلغاء إسناد عميل لمندوب (`sales_rep_customers`) | **مُغطّاة الآن** (PILOT-OPS-AUDIT-005، أعلاه) — مع إغلاق الوصول العالق وتطهير دورة الحياة |
| إبطال جلسة/JWT لعضو مُزال | **خارج النطاق** (الجلسة تبقى حتى انتهائها) |
| قرار طلب تسجيل المحل (قبول/رفض) | **مُغطّى الآن** (PILOT-OPS-AUDIT-006، أعلاه) — القبول يُصدر `customer.created` + `customer_signup_request.approved`؛ الرفض يُصدر `customer_signup_request.rejected` |
| تقديم طلب التسجيل نفسه (الإرسال المجهول) | **بلا حدث — مقصود** (المُقدِّم مجهول بلا فاعل؛ صف الطلب هو سجل التقديم) |

**التصنيف للتجربة المراقَبة:** الفجوات المتبقية **غير حاجبة (P3)** — أساس التدقيق يغطّي دورات
حياة الزبون والطلب والمنتج **وإعداد/تهيئة المخزون وعضوية/دعوات/أدوار الفريق وإعدادات المنشأة/
المنطقة الزمنية/الضريبة وإسناد المندوبين وقرار طلب تسجيل المحل**، وتغيّرات الكمية محفوظة في سجل
الحركة (مع الفاعل الآن)، وحركات الطلبات غير مُكرَّرة. الاستثناء المتبقّي **مقصود**: تقديم طلب
التسجيل نفسه (الإرسال المجهول) بلا فاعل، وصف الطلب هو سجل التقديم. لا حاجب P0/P1/P2 هنا.

### القسم 10-ب — مصفوفة الاستجابة للحوادث (Incident Response Matrix)

> الجدول أعلاه يحدّد **ما هي** الشدّة. هذا الجدول يحدّد **من يستجيب وكيف**.
> الأسماء الفعلية تُملأ في جدول ملكية الدعم (القسم 12-أ) — لا تُدوَّن هويات في هذا المستودع.

| الشدّة | المسؤول | هدف الإقرار | الاحتواء الفوري | الأدلة المطلوبة | التواصل | حالة التجربة | اعتماد الحل |
|---|---|---|---|---|---|---|---|
| **P0** | المسؤول التقني + مالك النشاط | **فورًا** | أوقِف الوصول المتأثّر؛ ألغِ `customer_access_links` عند تسريب رابط؛ دوّر الأسرار عبر مزوّدها | احفظ الصفوف وأثر التدقيق — **لا حذف** | إبلاغ جهة النشاط **فورًا** | **STOP — إيقاف التجربة** | Control Room |
| **P1** | المسؤول التقني | **خلال ساعة** ضمن ساعات الدعم | علِّق سير العمل المتأثّر | سجلات + `/api/health` + `migration list` | إبلاغ جهة النشاط | **PAUSE — تعليق حتى الحل** · قرار تقني في **نفس يوم العمل** | Control Room |
| **P2** | المسؤول التقني | **يوم العمل التالي** | وثّق الحلّ البديل | مراجع الطلبية/الزبون المتأثّرة | إبلاغ جهة الاتصال | استمرار **فقط** بحلّ بديل محدود مقبول + موافقة المسؤول التقني | المسؤول التقني |
| **P3** | المسؤول التقني | أفضل جهد | لا شيء | تُدوَّن في سجل P3 | لا يلزم | استمرار | المسؤول التقني |

**قواعد ملزمة:**

- **تجميد النشر** يسري عند P0 و P1.
- **إيقاف الطلبيات الجديدة** يسري عند P0، وعند P1 إذا كانت سلامة الطلبيات موضع شك.
- **مراجعة ما بعد الحادث إلزامية** لكل P0 و P1.
- ⚠️ **استثناء مهم:** **طلبية مكرّرة غير مُفسَّرة** أو **تباين مخزون غير مُفسَّر** **لا يبقى P2 بحلّ بديل**.
  يُعامَل بوصفه **PAUSE** حتى تُفهَم سلامة البيانات ويُحدَّد السبب الجذري. عدم الفهم ليس حلًّا بديلًا.
  - قبل تصنيفه، **اسأل صراحةً:** هل كانت هناك أكثر من علامة تبويب مفتوحة عند الإرسال؟
    طلبيتان من تبويبين منفصلين هما **محاولتان منطقيّتان منفصلتان ومتوقّعتان**، وليستا خللًا في
    منع التكرار بقاعدة البيانات (راجع القسم 7 والقسم 5-ب).

## القسم 11 — خطة التراجع والحوادث (Rollback / Incident)

> **لا تعليمات إعادة تعيين مُدمّرة هنا.** شجرة قرار آمنة فقط.

### قاعدة عامة: التصحيح الأمامي مُفضَّل (Forward-fix first)

> عندما تكون قاعدة البيانات قد **تقدّمت بأمان** (هجرة إضافية نجحت، والبيانات سليمة)، فالاتجاه الصحيح
> هو **الأمام**: انشر التطبيق المتوافق. إرجاع الكود إلى الخلف تحت مخطط أحدث هو الخيار **الأخطر**.

| الحالة | الإجراء الصحيح |
|---|---|
| فشل نشر التطبيق **قبل** أي هجرة | أعِد نشر نشرة Vercel سبق التحقّق منها — المخطط لم يتحرّك |
| **الهجرة طُبِّقت والتطبيق لم يُنشر** (تباين قاعدة-أولًا) | **تصحيح أمامي: انشر التطبيق.** لا تُرجِع المخطط للخلف |
| نشر تطبيق غير متوافق مع المخطط المستضاف | حدّد الاتجاه أولًا؛ إن كان المخطط **أحدث** فانشر للأمام. لا تُرجِع الكود إلا إذا ثبت توافقه مع المخطط الأحدث |
| **نتيجة هجرة غامضة** | **قِف.** لا تُعِد التشغيل، لا تُصلِح السجل. اجمع دليل `supabase migration list --linked` وصعّد |
| فشل الدخول (Auth) | افحص إعداد المستضاف أولًا (`MADAF_AUTH_PRIMARY_METHOD`، مزوّد SMS) قبل لمس الكود — الأرجح إعداد لا نشر |
| فشل صلاحية دالة الانضمام (onboarding) وقت التشغيل | تصحيح أمامي مُراجَع فقط |
| فشل منع تكرار الطلبيات (idempotency) | احفظ صفوف المطالبة كدليل — **لا تُعدّل `order_submission_claims` يدويًا** |
| تباين مخزون | جمّد سير العمل المتأثّر، احفظ أثر التدقيق، صحّح بهجرة أمامية مُراجَعة |
| فشل التخزين/الصور | افحص أولًا وجود إعداد توقيع الصور الخادمي — ظهور صور بديلة هو العرَض المتوقّع لغيابه |
| فشل إنشاء مستند/PDF | لا يؤثّر على الطلبية نفسها؛ صحّح للأمام |

**⛔ لا يُوصى بأيٍّ ممّا يلي تلقائيًا — كلٌّ منها يحتاج موافقة Control Room صريحة:**

- ❌ `force-push` لنسخة `main` أقدم (خصوصًا بعد تطبيق هجرة).
- ❌ إعادة تعيين قاعدة البيانات المستضافة (`db reset`).
- ❌ إصلاح سجل الهجرات (`migration repair`) كإجراء روتيني.
- ❌ تشغيل الهجرة مرّتين.
- ❌ إسقاط كائنات مخطط جديدة.
- ❌ حذف صفوف بيانات عمل.
- ❌ تعديل صفوف `order_submission_claims`.
- ❌ تعديل سجل التدقيق (`audit_events`).

> **يتطلّب موافقة Control Room الصريحة:** أي تعديل لبيانات الإنتاج · أي استعادة (restore) ·
> أي تراجع (rollback) · أي هجرة مستضافة · أي تغيير متغيّر بيئة · أي إصلاح لسجل الهجرات.

**تراجع على مستوى التطبيق فقط (كود):**
- أوقِف انضمام أي نشاط تجريبي جديد إذا لزم.
- حدِّد آخر نشر إنتاج معروف جيّد.
- **فضّل التراجع/إعادة النشر عبر Vercel** لنشرة سبق التحقّق منها.
- **تأكّد من توافق قاعدة البيانات** قبل إرجاع الكود للخلف (الكود الأقدم مع مخطط أحدث).

**مشكلة هجرة قاعدة بيانات:**
- **لا `db reset`** على المستضاف.
- **لا migration repair** باستهتار.
- **لا تعدّل هجرات تاريخية.**
- أوقِف الكتابات المتأثّرة مؤقتًا.
- افحص هل الهجرة **إضافية ومتوافقة رجعيًا** (هجرة C2 إضافية: تعريف دالة + قيد `NOT VALID`).
- عند الحاجة، أنشئ **هجرة تصحيح أمامية مُراجَعة** (forward-fix)، لا تصحيحًا يدويًا.

**مشكلة سلامة بيانات:**
- **احفظ الأدلة** (لا تحذف/تكتب فوق السجلات).
- حدِّد المستأجر والصفوف المتأثّرة.
- أوقِف العملية المتأثّرة.
- **لا حذف/كتابة يدوية** بلا خطة تصحيح مُراجَعة.
- احفظ أثر سجل النشاط/التدقيق.

**مشكلة أمنية:**
- أوقِف الوصول المتأثّر.
- دوّر الأسرار المنكشفة **عبر مزوّدها الصحيح** (لوحة Supabase / Vercel).
- أبطِل بيانات الاعتماد/الجلسات المخترَقة حيث يُدعم (مثلًا **إلغاء `customer_access_links`** لقطع وصول الرموز).
- وثّق الزمن والنطاق.

**محظورات صريحة (لا تفعل):**
- ❌ لا `db reset` على المستضاف.
- ❌ لا `force-push` إلى `main`.
- ❌ لا حذف يدوي لبيانات التجربة.
- ❌ لا استخدام `service_role` في كود المتصفّح/العميل.
- ❌ لا إصلاحات SQL غير مُراجَعة.

### النسخ الاحتياطي والاستعادة الزمنية (Backup / PITR) — **تحقّق يدوي مستضاف مطلوب**

قبل GO، يجب أن يؤكّد المهندس **يدويًا على لوحة Supabase** (هذا الدليل توثيقي — لا يتّصل بالمستضاف
ولا يفعّل شيئًا):

- [ ] النسخ الاحتياطية التلقائية اليومية **مفعّلة** للمشروع المستضاف.
- [ ] **الاستعادة الزمنية (PITR)** — حالتها **مؤكَّدة صراحةً** (مفعّلة/غير متاحة)، ونافذة الاستبقاء مُسجَّلة.
- [ ] المهندس يعرف **من أين** يبدأ استعادة، و**من** يأذن بها (مُعتمِد الاستعادة، القسم 12-أ).
- [ ] **RPO ≤ 24 ساعة** و**RTO ≤ 8 ساعات** مقبولان ومُسجَّلان.
- [ ] لا اعتماد على `db reset`/حذف يدوي كخطة تعافٍ — التعافي عبر نسخة/PITR فقط، أو هجرة تصحيح
      أمامية مُراجَعة.

> **الإجراء الكامل والسجل القابل للتعبئة في وثيقة منفصلة:**
> [`BACKUP_RESTORE_AND_RECOVERY.md`](BACKUP_RESTORE_AND_RECOVERY.md).
> هذا القسم يضع البوابة؛ تلك الوثيقة تُنفِّذها وتحفظ الدليل.

> **الحالة الحالية: `BACKUP/PITR STATUS — MANUAL HOSTED VERIFICATION REQUIRED`.**
> إن تعذّر تأكيد النسخ الاحتياطي/PITR على المستضاف، فهذا **حاجب NO-GO** حتى يُؤكَّد.
> **إذا كانت PITR غير متاحة:** NO-GO ما لم تمنح غرفة التحكّم **استثناء مخاطر خطّيًا ومنفصلًا** لاحقًا —
> **هذه المرحلة التوثيقية لا تمنح ذلك الاستثناء.**

### تسلسل النشر الآمن (Deployment Order) — للمهندس، لا يُنفَّذ من هذا الدليل

عند نشر المراحل المحلية (1–8) على المستضاف، بالترتيب، مع تحقّق بعد كل خطوة:

1. **البناء أخضر محليًا:** `npm run build` (يشغّل `check:deploy-safety` + `next build` + فحص المسارات
   الديناميكية) و`npm run lint` و`npm test` — كلها نظيفة.
2. **الهجرات أولًا ثم التطبيق:** ادفع الهجرات إلى المستضاف (`supabase db push --linked --yes`) **قبل**
   نشر الكود الذي يعتمدها، فلا يعمل كود أحدث على مخطط أقدم. (تحذير ذاكرة شهادة pg-delta بعد الدفع
   معروف وغير حاجب.)
3. **تحقّق التطابق:** `supabase migration list --linked` — بلا معلّق (المحلي == المستضاف).
4. **انشر التطبيق** عبر Vercel من `main` بعد الدمج المصرّح به.
5. **فحص حيّة بعد النشر:** `GET /api/health` يُعيد `status: ok` و`environment: production`، وحقل
   `commit` (**سبعة محارف**) يطابق `git rev-parse --short=7 main` — راجع «إجراء تثبيت خط الأساس» (القسم 1).
6. **دخّن المسارات الحرجة:** الدخول، `/admin/*` محميّة، لوحة التحكم/المنتجات/الطلبيات/المخزون/
   المستندات تفتح، ولا انكشاف عبر المستأجرين. **نفّذ فحوص الدخان في مستأجر `PILOT-SMOKE` فقط**
   (القسم 5-ب) — لا في مستأجر النشاط الحقيقي.
7. **تذكير:** `/api/health` **لا يفحص قاعدة البيانات**. اتصال قاعدة البيانات يُثبَت فقط بفتح شاشة
   تعتمد بيانات فعلية (المنتجات/الطلبيات) داخل التطبيق.

## القسم 11-ب — تجميد التغيير أثناء التجربة (Pilot Change Freeze)

> **قيمة التجربة المراقَبة كلّها تقوم على نسبة كل ملاحظة إلى بناء واحد معروف.** أي تغيير غير مُتحكَّم به
> يُفسد ذلك.

| البند | القيمة |
|---|---|
| **بداية التجميد** | لحظة **توقيع GO** (القسم 13) |
| **نهاية التجميد** | اكتمال **إغلاق الأسبوع الأول** (القسم 14) |

**ممنوع أثناء التجميد:**
- ❌ أي عمل ميزات اعتيادي.
- ❌ أي تغيير راحة من فئة P2 أو P3.

**مسموح أثناء التجميد:**
- ✅ تصحيحات طوارئ **P0 / P1** فقط.
- ✅ تصحيح **P2** **فقط** عندما تصنّفه غرفة التحكّم صراحةً بأنه **حاجب للتجربة**.

**كل تغيير طارئ يتطلّب — بلا استثناء:**

- [ ] **مُعتمِد إصدار مُسمّى** (القسم 12-أ).
- [ ] **فرع معزول** (لا عمل مباشر على `main`).
- [ ] **أصغر فرق ممكن** (minimal diff) — لا تنظيف مصاحب، لا إعادة هيكلة.
- [ ] **تصنيف مخاطر** مكتوب.
- [ ] **الاختبارات ذات الصلة** خضراء.
- [ ] **مراجعة مستقلة** بعمق يتناسب مع المخاطر.
- [ ] **فحص هجرة مُسبق (preflight)** عند وجود هجرة.
- [ ] **موافقة صريحة على دفع قاعدة البيانات** عند وجود هجرة.
- [ ] **دمج fast-forward** فقط.
- [ ] **تحقّق صحّة الإنتاج** بعد النشر (`/api/health` + تطابق خط الأساس).
- [ ] **تحقّق يدوي موجَّه** لسير العمل المتأثّر.
- [ ] **إبلاغ جهة اتصال النشاط** بالتغيير وأثره.

> **لا تراجع تلقائي لمخطط تقدّم.** إذا كانت قاعدة البيانات قد تقدّمت بأمان، فالمسار هو التصحيح الأمامي
> (القسم 11) — لا إرجاع الكود تحت مخطط أحدث.

## القسم 13 — قائمة الإطلاق GO / NO-GO

**يتطلّب GO تحقّق كل ما يلي:**

**(أ) خط الأساس والنشر**
- [ ] **تطابق خط الأساس** وفق القسم 1: `git rev-parse --short=7 main` == حقل `commit` من `/api/health` (**مقارنة نصّية تامة، سبعة محارف**).
- [ ] `/api/health` يُعيد `status: ok` و`environment: production` و`service: madaf`.
- [ ] Vercel في حالة Ready، والفرع المنشور `main`.
- [ ] هجرات Supabase متطابقة، بلا معلّق (`supabase migration list --linked` — المحلي == المستضاف؛ أحدثها `20260812100000_operational_hardening.sql`).
- [ ] **لا يُوقَّع GO على commit تطبيق يخالف خط أساس قاعدة البيانات.** إن اختلفا: NO-GO.

**(ب) النسخ الاحتياطي والتعافي — بوابة حاجبة**
- [ ] سجلّ التحقّق في [`BACKUP_RESTORE_AND_RECOVERY.md`](BACKUP_RESTORE_AND_RECOVERY.md) **مكتمل** ونتيجته:
      **`BACKUP/PITR VERIFIED — GO ELIGIBLE`**.
- [ ] أي من التالي يعني **NO-GO**: خطة المشروع غير معروفة · حالة النسخ الاحتياطي غير معروفة ·
      مدة الاستبقاء غير معروفة · حالة PITR غير مؤكَّدة · نقطة بدء الاستعادة غير معروفة ·
      لا مُعتمِد استعادة مُعيَّن · RPO/RTO غير مقبولين · لا دليل مُسجَّل.

**(ج) الدخول عبر Phone OTP — بوابة حاجبة**
- [ ] سجلّ فحص Phone OTP (القسم 13-ج أدناه) نتيجته: **`PHONE OTP VERIFIED — GO ELIGIBLE`**.
- [ ] النتيجة **`PHONE OTP UNVERIFIED — NO-GO`** تعني إيقاف الإطلاق والرجوع إلى غرفة التحكّم.

**(د) الملكية التشغيلية**
- [ ] كل حقول جدول «ملكية الدعم» (القسم 12-أ) **مُعبَّأة** في النسخة التشغيلية — لا يُوقَّع GO مع حقل فارغ.
- [ ] مُعتمِد الإصدار، مُعتمِد الحوادث، ومُعتمِد الاستعادة **مُعيَّنون بالاسم** في النسخة التشغيلية.
- [ ] المستأجر التشغيلي `PILOT-SMOKE` جاهز ومنفصل عن مستأجر النشاط الحقيقي (القسم 5-ب).

**(هـ) سير العمل**
- [ ] تسجيل الدخول يعمل.
- [ ] سياق المستأجر صحيح.
- [ ] دور مالك التجربة صحيح.
- [ ] المحلات تفتح.
- [ ] المنتجات تفتح.
- [ ] الطلبيات تفتح.
- [ ] المخزون يفتح.
- [ ] صورة منتج مرفوعة **تظهر فعليًا** في `/shop/<token>` (وليست صورة بديلة) — هذا هو كاشف إعداد توقيع الصور.
- [ ] الروابط العامة تستخدم الرابط القياسي.
- [ ] لا حاجب معروف P0/P1/P2.
- [ ] عملية/جهة التراجع مفهومة.
- [ ] سجل الدعم جاهز.

**أمثلة NO-GO:**
- عدم تطابق خط الأساس (`--short=7 main` ≠ حقل `commit`).
- فشل تطابق الهجرات.
- **`BACKUP/PITR UNVERIFIED`** — أو PITR غير متاح بلا استثناء خطّي لاحق من غرفة التحكّم.
- **`PHONE OTP UNVERIFIED`** — تعذّر إرسال أو تحقّق رمز OTP على رقم اختبار حقيقي.
- حقول ملكية الدعم غير مُعبَّأة.
- انكشاف بيانات عبر المستأجرين.
- تناقض قبول/رفض التسجيل.
- وصول دور غير مصرّح.
- تعذّر سير العمل الأساسي.
- انكشاف سرّ خام.
- تغيّر بيانات غير مُفسَّر.
